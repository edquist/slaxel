(10:55:39) tannenba: IIRC, the starter file transfer used to block both on transfer of input and output. We wanted the the starter to be responsive because if the slot is preempted, we wanted to go away quickly so a) the slot is free and b) the job can be rescheduled somewhere else. Previously, when transfer inputs could take 20+ min, it was all just badput since the job will never actually be started. For transfer output, however, we don't want to be responsive to preemptions.... we want to try and send the job output back right until the moment we are killed to avoid making the entire job run badput. Thus the decision that it was not very important to do the work to make transfer output happen in a forked child...
